# CrabOnTree Architecture

This document describes the architecture of CrabOnTree, including its layered design, data flow, and threading model.

## Overview

CrabOnTree is built using a clean, layered architecture that separates concerns and enables testability, maintainability, and future extensibility.

```
┌─────────────────────────────────────────┐
│         UI Layer (ui_egui)              │
│  - egui application                     │
│  - User interaction                     │
│  - Theme application                    │
└────────────┬────────────────────────────┘
             │
             ▼
┌─────────────────────────────────────────┐
│      UI Core Layer (ui_core)            │
│  - Framework-agnostic types             │
│  - Themes and colors                    │
│  - Keyboard shortcuts                   │
└─────────────────────────────────────────┘
             │
             ▼
┌─────────────────────────────────────────┐
│      Application Layer (app)            │
│  - State management (Elm architecture)  │
│  - Message passing                      │
│  - Effect handling                      │
│  - Async job executor                   │
│  - Configuration management             │
└────────────┬────────────────────────────┘
             │
             ▼
┌─────────────────────────────────────────┐
│        Git Layer (git)                  │
│  - gitoxide wrapper                     │
│  - Repository operations                │
│  - Error handling                       │
└─────────────────────────────────────────┘
```

## Layer Responsibilities

### Git Layer (`crabontree-git`)

**Purpose**: High-level wrapper around gitoxide (gix) for Git operations.

**Key Types**:
- `GitRepository`: Main repository interface
- `GitError`: Error types for Git operations
- `StatusSummary`: Working directory status

**Responsibilities**:
- Opening repositories
- Reading branches and HEAD
- Getting repository status
- Error translation from gitoxide to application errors

**Dependencies**: gitoxide (gix), thiserror, tracing

**Testing**: Integration tests with temporary Git repositories

### Application Layer (`crabontree-app`)

**Purpose**: Core business logic, state management, and orchestration.

**Key Types**:
- `AppState`: Complete application state
- `AppMessage`: Events that trigger state changes
- `Effect`: Side effects generated by state changes
- `Job`: Async operations
- `JobExecutor`: Background job runner

**Responsibilities**:
- State management using Elm-like architecture
- Message-based state updates (pure reducer function)
- Effect execution (side effects)
- Async job scheduling and execution
- Configuration persistence

**Dependencies**: Git layer, tokio, directories, serde, toml

**Testing**: Unit tests for reducer, integration tests for job executor

### UI Core Layer (`crabontree-ui-core`)

**Purpose**: Framework-agnostic UI primitives.

**Key Types**:
- `Color`: RGBA color with hex parsing
- `Theme`: Color scheme definition
- `Shortcut`: Keyboard shortcut bindings
- `Action`: Application actions

**Responsibilities**:
- Color management and parsing
- Theme definitions (dark/light)
- Keyboard shortcut definitions
- No framework-specific code

**Dependencies**: serde (minimal)

**Testing**: Unit tests for color parsing and theme creation

### UI Layer (`crabontree`)

**Purpose**: egui-based user interface.

**Key Types**:
- `CrabOnTreeApp`: Main application struct implementing `eframe::App`

**Responsibilities**:
- Rendering UI with egui
- Polling messages from async jobs
- Converting messages to state updates
- Executing effects (submitting jobs, saving config)
- Applying themes to egui
- User interaction handling

**Dependencies**: App layer, UI Core layer, eframe, egui, rfd

**Testing**: Manual testing (Phase 0)

## Data Flow

### User Action to State Update

```
User Action
    │
    ▼
UI Event Handler
    │
    ▼
AppMessage Created
    │
    ▼
Reducer Function (pure)
    │
    ├─> State Update
    │
    └─> Effect Generated
         │
         ▼
    Effect Execution
         │
         ├─> Job Submission
         ├─> Config Save
         └─> Batch Effects
```

### Async Job Execution

```
Effect::OpenRepo(path)
    │
    ▼
JobExecutor::submit(Job::OpenRepo)
    │
    ▼
Worker Thread (tokio runtime)
    │
    ▼
spawn_blocking (Git operations)
    │
    ├─> GitRepository::open()
    ├─> GitRepository::get_head()
    ├─> GitRepository::get_branches()
    └─> GitRepository::get_status()
    │
    ▼
Result → AppMessage::RepoOpened
    │
    ▼
Bounded Channel (capacity: 100)
    │
    ▼
UI Thread (try_recv)
    │
    ▼
Reducer (state update)
```

## Threading Model

### Main Thread (UI Thread)

**Responsibilities**:
- egui rendering (60 FPS)
- Message polling (`try_recv()` - non-blocking)
- State updates via reducer
- Effect execution (job submission, config save)

**Characteristics**:
- Never blocks on I/O
- Polls messages every 100ms
- Fast, responsive UI

### Worker Thread

**Responsibilities**:
- Runs tokio runtime
- Processes jobs from unbounded queue
- Executes Git operations in `spawn_blocking`
- Sends results to UI via bounded channel

**Characteristics**:
- Long-lived (entire application lifetime)
- Handles backpressure via bounded result channel
- Graceful error handling (doesn't crash on errors)

### Channel Architecture

```
UI Thread                Worker Thread
    │                         │
    │  Job Submission         │
    ├───────────────────────>│
    │  (unbounded)            │
    │                         │
    │                         ├─> Execute Job
    │                         │   (spawn_blocking)
    │                         │
    │  Result Messages        │
    │<────────────────────────┤
    │  (bounded: 100)         │
    │                         │
    ▼                         ▼
```

**Unbounded Job Channel**: UI never blocks when submitting jobs (user events are infrequent)

**Bounded Result Channel**: Prevents memory issues if UI can't keep up with job results

## Message Flow

### Message Types

**User-Triggered**:
- `OpenRepoRequested(path)`
- `CloseRepo`
- `RefreshRepo`
- `ClearError`

**Job Results**:
- `RepoOpened { path, head, branches, status }`
- `RepoRefreshed { head, branches, status }`
- `Error(string)`

### State Transitions

```
Initial State (no repo open)
    │
    │ OpenRepoRequested(path)
    ├──> loading = true
    ├──> Effect::OpenRepo(path)
    │
    │ [Job executes in background]
    │
    │ RepoOpened { ... }
    ├──> loading = false
    ├──> current_repo = Some(...)
    ├──> Add to recent repos
    └──> Effect::SaveConfig

Repository Open
    │
    │ RefreshRepo
    ├──> loading = true
    ├──> Effect::RefreshRepo(path)
    │
    │ [Job executes in background]
    │
    │ RepoRefreshed { ... }
    ├──> loading = false
    └──> Update repo data

    │
    │ CloseRepo
    ├──> current_repo = None
    └──> Effect::None

Any State
    │
    │ Error(message)
    ├──> loading = false
    └──> error = Some(message)
```

## Error Handling

### Error Flow

```
Git Operation
    │
    ▼
GitError (thiserror)
    │
    ▼
Wrapped in anyhow::Error with context
    │
    ▼
Converted to AppMessage::Error(string)
    │
    ▼
State update: error = Some(...)
    │
    ▼
UI displays error with dismiss button
```

### Error Types

**Git Layer**: `GitError` enum with specific variants
- `RepoNotFound`
- `InvalidRepo`
- `OperationFailed`
- `RefNotFound`

**Application Layer**: `anyhow::Error` for flexibility and context chaining

**UI Layer**: Error strings for display

## Configuration

### File Location

Platform-specific configuration directory:
- **Linux**: `~/.config/crabontree/CrabOnTree/config.toml`
- **macOS**: `~/Library/Application Support/com.crabontree.CrabOnTree/config.toml`
- **Windows**: `%APPDATA%\crabontree\CrabOnTree\config\config.toml`

### Format

TOML format with serde:

```toml
theme = "dark"
max_recent = 10

[[recent_repos]]
"/home/user/project1"

[[recent_repos]]
"/home/user/project2"
```

### Loading Strategy

1. Attempt to load from disk
2. On error (not found, parse error, etc.): Use defaults and log warning
3. Never crash on config errors

### Saving Strategy

1. Create parent directories if needed
2. Serialize to TOML
3. Write atomically to disk
4. Log errors but don't crash

## Testing Strategy

### Unit Tests

- **Git Layer**: Temporary repositories with git CLI
- **App Layer**: Reducer logic, message handling
- **UI Core**: Color parsing, theme creation
- **UI Layer**: Manual testing (Phase 0)

### Integration Tests

- Git operations with real repositories
- Job executor with job submission and results
- Configuration persistence

### Test Isolation

- Each test creates its own temporary directory
- No shared state between tests
- Tests can run in parallel

## Future Extensions

### Phase 1: Commit History

- Add `CommitInfo` type to Git layer
- Add `GetCommits` job
- Add commit list to UI

### Phase 2: Diffs and Staging

- Expand `StatusSummary` with file-level details
- Add staging/unstaging operations
- Add diff viewer

### Phase 3: Commit Creation

- Add commit message editor
- Add staging area UI
- Add commit operation

### Architectural Benefits

- **Layered design**: Changes in one layer don't affect others
- **Message passing**: Easy to add new operations
- **Async jobs**: UI never freezes
- **Pure reducer**: Deterministic, testable state updates
- **Effect system**: Clear separation of logic and side effects
- **Framework-agnostic UI core**: Can add other frontends (iced, etc.)
